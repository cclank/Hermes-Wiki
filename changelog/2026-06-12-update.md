## 1. Subagent delegation — 默认移除墙钟超时，由心跳 staleness 监控替代（架构主轴 A）

### 1.1 `bba9b519` `fix(delegation): remove the default subagent wall-clock timeout (#45149)`

`2026-06-12T12:58:25 -0700` teknium1。

**问题**：进行合法重活的子 agent（深度 code review / 研究 fan-out / 慢 reasoning 模型）经常在已成功完成 36 次 API 调用、仍在稳步推进时，被毯子式 600s `child_timeout_seconds` 一刀切。失败信号应来自子 agent 实际在干的事——API 错误、工具错误、迭代预算——而不是 delegation 级别的秒表。

**改动**（`tools/delegate_tool.py`）：

- `tools/delegate_tool.py:567` `DEFAULT_CHILD_TIMEOUT: Optional[float] = None` — 之前 600（秒）。
- `tools/delegate_tool.py:400-439` `_get_child_timeout()` 在用户未配置时返回 `None`；`delegation.child_timeout_seconds` 配置项与 `DELEGATION_CHILD_TIMEOUT_SECONDS` 环境变量均允许 opt back in，floor 30s（`return None if parsed <= 0 else max(30.0, parsed)` 双层 guard 在 config + env 两路）。
- `Future.result(timeout=None)` 等子代理自然结束，不再 raise `TimeoutError`。

**Stuck-child 保护改由 heartbeat staleness 监控承担**（`tools/delegate_tool.py:577-578`）：

- `_HEARTBEAT_STALE_CYCLES_IDLE = 15`（15 × 30s = 450s）：无 `current_tool` 时连续 15 个心跳周期无 API-call 进展 → 视为 idle stale → 停止刷新 parent activity → gateway 闲置超时接管。
- `_HEARTBEAT_STALE_CYCLES_IN_TOOL = 40`（40 × 30s = 1200s）：有 `current_tool` 时连 40 周期无进展 → 视为 in-tool stale。In-tool ceiling 远高于 idle，因合法 long-running 工具（terminal 命令、web fetch、大文件读）需要充足时间完成。

**docs 与默认对齐**：
- `website/docs/user-guide/features/delegation.md:186-187`：`child_timeout_seconds: 0     # default: 0 = no timeout` + 注释行 `# child_timeout_seconds: 1800  # opt-in hard cap (floor 30s)`。
- `website/i18n/zh-Hans/.../delegation.md:186-187` 同步中文版。
- `hermes_cli/config.py:1763`：`"child_timeout_seconds": 0,  # optional wall-clock cap per child agent. 0 (default)`.
- `AGENTS.md:977` 文档级 mention 不变。

**配置兼容性**：旧 `child_timeout_seconds: 600` 等正数配置自动按 `max(30.0, parsed)` 继续生效，无 breaking change，仅默认值翻转。

## 2. Dashboard skill install 静默自取消修复（dashboard 安装按钮端到端可用）

### `a118b94a` `fix(dashboard): skill installs from the dashboard silently auto-cancel (#45150)`

`2026-06-12T12:58:36 -0700` teknium1（HEAD）。

**问题**：dashboard 的 `/api/skills/hub/install`（及新建 profile 的 `hub_skills` 路径）spawn `hermes skills install <id>` 时 `stdin=DEVNULL` 但**未传 `--yes`**。`do_install()` 的 `Confirm [y/N]` prompt 触 EOF → 默认 `n` → 打印 `'Installation cancelled.'` 到用户**永远看不到**的后台日志——每个 dashboard 安装无操作。

**修**（`hermes_cli/web_server.py`）：
- `web_server.py:8108` `install_skill_hub`：`_profile_cli_args(body.profile or profile) + ["skills", "install", identifier, "--yes"]` — 末尾追加 `--yes`。
- `web_server.py:8847` `create_profile_endpoint` 的 `hub_installs` 循环：`["-p", body.name, "skills", "install", ident, "--yes"]` — 同步加。

理由：dashboard 安装按钮**就是**显式用户同意，对齐已经传 `--yes` 的 uninstall 端点，以及 TUI / slash-command `skip_confirm` 的相同 rationale。

**复现**：`stdin=DEVNULL` 的精确 argv 对临时 `HERMES_HOME` 重放——无 `--yes` 取消，加上后真装上。

## 20. Update — system-scope hermes-gateway 重启不再触发 polkit 交互（#45145）

### `9b01c4d1` `fix(update): never spawn an interactive polkit prompt when restarting a system-scope gateway (#45145)`

`2026-06-12T12:38:15 -0700` teknium1。

**问题**：`hermes update` 重启 system-scope `hermes-gateway` 服务时，作为 non-root 用户调用 `systemctl reset-failed / start / restart` → polkit 的 `org.freedesktop.systemd1.manage-units` TTY auth agent 弹出。该 prompt 在 captured subprocess 内跑（10-15s timeout），user 看到它**闪现即死**还没来得及回答；resulting `TimeoutExpired` 被循环的**blanket `except`** 静悄悄吞 —— restart phase 直接消失无输出，update 看似成功但 gateway 没真重启。

**修**：
- 启动前 resolve `manage-units` 命令 prefix：plain `systemctl` as ...（commit message 截断处，但实质是探测 — 当 non-root 且 unit scope 是 `--system` → 改走 `sudo -n systemctl ...` non-interactive sudo，sudo 拒就直接 fail-fast；user-scope 则用 plain `systemctl --user`）。
- Polkit prompt 路径绕过 → restart phase 报真错误（"requires root, run with sudo" 等），不再静默吞。
